Lab: Test Bootc Containers With Anaconda

Estimated reading time: 5 minutes.

Objective

Deploy a bootc container image, from the local containers storage, in a local virtual machine (VM).

Before you Begin

You need a development machine running RHEL, to which you have access using an unprivileged user account.

These instructions were tested on RHEL 10.0 but should work with minimal or no changes on newer and older RHEL releases, since 9.6.

If you are using the course classroom, log in on the workstation VM as the user student with password student. If not, please adapt the instructions to your test environment.

Make sure you verified your test environment by following the first lab of this course.

Instructions

You will use the standard RHEL installation media, with a custom kickstart script, to install RHEL in a local VM from a bootc container image. Thanks to the direct kernel loading feature of libvirt, and also to virtio devices backed by user directories, you do NOT require PXE boot, HTTP servers, or custom installation media.

  1. Prepare to start this activity by checking that you have the outcomes from previous activities.

    1. If needed, create a local clone of the sample code repository of this course, but do NOT enter its directory. Stay on your home directory to start this lab.

      $ cd
$ git clone https://github.com/RedHatQuickCourses/rhde-bootc-samples.git
Cloning into 'rhde-bootc-samples'...
...

    2. If needed, login to your private registry.

      $ podman login -u student -p redhat registry.lab.example.com:5000
Login Succeeded!

    3. Verify that you have a bootc container image named localhost/httpd-bootc in your local container storage. If you do not, please perform the previous lab, which builds that image.

      $ podman image list | grep httpd-bootc
localhost/httpd-bootc                            latest      dc263eabb8d7  18 hours ago  1.56 GB

    4. Verify that you have access to libvirt through its session interface. If you do not, please perform the preparation tasks from the first lab.

      $ virsh list --all
 Id   Name   State
--------------------
      It is fine if you have some local VMs on your system, as long as there isn’t a VM named bootc-test.

  2. Prepare scripts and configuration files you will use to automate creation of a test VM.

    1. Copy your bootc container image to a local directory, in a format that Anaconda can use as its installation source.

      Make sure you create your temporary work directories on your home directory instead of inside the clone of the samples repository. 
      $ cd
$ mkdir temp-bootc
$ skopeo copy containers-storage:localhost/httpd-bootc oci:temp-bootc/oci:httpd-bootc

    2. Create a working directory and copy the sample files to it. Then make the temporary working directory your current directory.

      $ mkdir temp-test-vm
$ cp rhde-bootc-samples/ks/inst.ks temp-test-vm
$ cp rhde-bootc-samples/ks/virt-install.sh temp-test-vm
$ cd temp-test-vm

    3. Review the example kickstart file named inst.ks.

      It is a standard RHEL Anaconda configuration for unattended installation, which uses a number of features:

      1. A virtiofs mount, in a %pre script, to get the contents of your bootc container image.

      2. The ostreecontainer command, to set a bootc container image as the installation source.

      3. A %post script which creates an initial user, with a known password and SSH key.

      4. Also in the %post script, it sets an initial hostname, which helps you identify your test VM and recognize it didn’t go into any day-2 configuration or on-boarding process yet.

        %pre --erroronfail
mkdir -p /mnt/host-var-srv
mount -t virtiofs -o ro host-var-srv /mnt/host-var-srv
%end

lang en_US.UTF-8
keyboard us
timezone Etc/UTC --utc
text

zerombr
clearpart --all --initlabel
reqpart --add-boot
part / --grow --fstype xfs

rootpw --lock
reboot

network --bootproto=dhcp

ostreecontainer --transport oci --url /mnt/host-var-srv/oci:httpd-bootc

%post --log=/var/log/anaconda/post-install.log --erroronfail

# local user for console login and SSH
useradd -g wheel core
echo "core:redhat123" | chpasswd
mkdir /home/core/.ssh
cat > /home/core/.ssh/authorized_keys << EOFSSH
REPLACE_WTH_SSH_PUB_KEY
EOFSSH

echo "bootc-test" > /etc/hostname
chmod 644 /etc/hostname
%end

    4. Create an SSH key for use with your test VM and embed it in the kickstart file.

      $ ssh-keygen -N '' -f edge-key -C 'initial key for edge devices'
Generating public/private rsa key pair.
$ SSH_PUB_KEY=$( cat edge-key.pub )
$ sed -i "s|REPLACE_WTH_SSH_PUB_KEY|$SSH_PUB_KEY|" inst.ks

    5. Review the virt-install.sh script which creates a test VM.

      It is designed to be run in text mode, so you do not need to switch to a graphical console or remote desktop session just to view the test VM boot screens. It is also designed to require no access to a remote container registry, by using the following features of libvirt:

      1. User mode networking with the Passt provider and port-forwarding to enable network connections from your development machine to services running inside the test VM.

      2. Direct kernel loading (the --location option) to grab the Linux kernel and initial RAM disk from a remote HTTP server.

      3. Injection of a custom kickstart file, so RHEL installation runs unattended.

      4. A virtiofs device (the --memorybacking and --filesystem options) to enable access to an installation source without remote network servers but from local host files.

      5. A serial console, so boot messages display in the current terminal, which could be an SSH session.

        #!/bin/bash
set -xeuo pipefail

virt-install \
--name bootc-test \
--vcpus 2 \
--ram 4096 \
--disk size=20 \
--network passt,portForward0=8022:22,portForward1=8080:80 \
--location http://content.example.com/rhel10.0/x86_64/dvd/ \
--os-variant rhel10.0 \
--initrd-inject ./inst.ks \
--memorybacking source.type=memfd,access.mode=shared \
--filesystem /home/student/temp-bootc,host-var-srv,driver.type=virtiofs \
--graphics none \
--extra-arg console=ttyS0 \
--extra-args inst.ks=file:/inst.ks

  3. Create a local VM from your bootc container images, using the provided scripts.

    1. Run the installation script.

      You could use it from a machine for which you do not have root access, as long as someone already installed Libvirt and its tooling.

      $ bash virt-install.sh

    2. Observe your test VM boot screen and RHEL Anaconda messages, as it boots, installs RHEL, and reboots into the newly installed system.

      The message, which states Deployment starting, indicates when Anaconda is done preparing the system and starts reading your bootc container image:

      ...
Setting up the installation environment
.
Configuring storage
Creating disklabel on /dev/vda
Creating xfs on /dev/vda3
Creating xfs on /dev/vda2
Creating biosboot on /dev/vda1
.
Running pre-installation scripts
.
Running pre-installation tasks
.
Installing the software
Deployment starting: /mnt/host-var-srv/oci:httpd-bootc
...

    3. If all goes well, you will see a message which states that your VM (domain or guest in libvirt’s jargon) was created, but you may not see its RHEL login screen on the terminal.

      ...
Domain creation completed.
Restarting guest.
Running text console command: virsh --connect qemu:///session console test-bootc
Connected to domain 'test-bootc'
Escape character is ^] (Ctrl + ])

    4. Detach from the VM console, by pressing Ctrl+], and start a SSH session to your test VM.

      $ ssh -i edge-key -p 8022 core@127.0.0.1
[core@bootc-test ~]$

    5. If there were issues, before you go back to fix your kickstart and script files, stop and destroy your test VM.

      Do NOT run these commands if all went well. You will still need your test VM in the next steps. 
      $ virsh destroy bootc-test
Domain 'bootc-test' destroyed
$ virsh undefine bootc-test --remove-all-storage
Domain 'bootc-test' has been undefined
Volume 'vda'(/home/student/.local/share/libvirt/images/bootc-test.qcow2) removed.

  4. Verify that your test VM was actually provisioned from the contents of your bootc container image.

    1. Verify that your test VM contains an active bootc deployment.

      The password of the core user (for sudo) is redhat123.

      $ sudo bootc status
● Booted image: oci:/mnt/host-var-srv/oci:httpd-bootc
        Digest: sha256:5ba977e2a75507b722535b21ae40f794176b61c63754e916a0531a6f9f76b894
       Version: 10.0 (2025-08-11 20:34:32.003406539 UTC)

    2. Verify that your test VM has an active Apache Web Server.

      $ rpm -q httpd
httpd-2.4.63-1.el10.x86_64
$ systemctl is-active httpd
active

    3. Verify that your test VM contains the files of your single-page web application.

      $ cat /var/www/html/index.html
This would be the HTML and JavaScript code of your single-page web application.

    4. Inspect the processes running inside your test VM.

      $ ps ax | wc -l
205
[core@bootc-httpd ~]$ pstree
systemd─┬─NetworkManager───3*[{NetworkManager}]
        ├─agetty
        ├─chronyd
        ├─dbus-broker-lau───dbus-broker
        ├─gssproxy───5*[{gssproxy}]
        ├─httpd─┬─httpd
        │       ├─httpd───68*[{httpd}]
        │       └─2*[httpd───52*[{httpd}]]
        ├─irqbalance───{irqbalance}
        ├─rhsmcertd───{rhsmcertd}
        ├─rpcbind
        ├─sshd───sshd-session───sshd-session───bash───pstree
        ├─systemd───(sd-pam)
        ├─systemd-hostnam
        ├─systemd-journal
        ├─systemd-logind
        ├─systemd-udevd
        └─systemd-userdbd───3*[systemd-userwor]

      Notice the much higher number of running processes, compared to the previous lab, where you run the same bootc container image from Podman. Also notice processes such as terminal login managers, a Cron daemon, and additional Systemd services you did not see running from Podman.

    5. Exit your SSH session and return to your development machine.

      $ exit
logout
Connection to 127.0.0.1 closed.

  5. Verify that your development machine can access the single-page web application running in the test VM.

    1. Verify that you get, through the libvirt port-forward tunnel, the contents of your single-page web application.

      $ curl 127.0.0.1:8080
This would be the HTML and JavaScript code of your single-page web application.

    2. If all went well, stop and destroy your test VM to conserve resources. If you want to keep it, you must edit your scripts to change the name of the VM you will create in the next activity.

      $ virsh destroy bootc-test
Domain 'bootc-test' destroyed
$ virsh undefine bootc-test --remove-all-storage
Domain 'bootc-test' has been undefined
Volume 'vda'(/home/student/.local/share/libvirt/images/bootc-test.qcow2) removed.
      You could reuse the same test VM, and use the bootc command to switch it to another container image from a remote container registry, but, for now, let’s ensure that we get a brand new test VM every time, without any state left over from previous tests.
